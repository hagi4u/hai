## Goals

- [ ] **리더십 전환**: 기술 결정 및 프로젝트 방향에서 "친절한 조율자"가 아닌 "명확한 결정권자"로 포지셔닝한다. (분기별 셀프 체크)
- [ ] **결단력 증명**: 반대 의견을 밀어내고 내 방향대로 진행해 성공한 사례를 **3건 이상** 만든다. → [사례 로그](#반대-의견-성공-사례-로그)
- [ ] **의사결정 속도**: 'Discussing' 상태 티켓을 **24시간 내 'Decided'** 로 전환하는 타임리밋을 스스로 건다.
- [ ] **안정성 PR**: KREAM Pay 프로덕션 오픈 시 **P0 이슈 0건**, 또는 MTTD(평균 감지 시간) **X시간 이내** 달성으로 가치 중심 성과를 만든다.
- [ ] **E2E 커버리지**: KREAM Pay Phase 2, 3의 핵심 결제 로직 E2E 테스트 커버리지를 최우선 과제로 삼고, **QA 리소스 30% 절감** 수치를 만든다.
- [ ] **도메인 오너십**: 스타일 또는 인증 도메인의 주담당자로 올라선다. (상반기 내 결정)
- [ ] **기술 공유 PR**: Tech Thursday에서 **분기 1회 이상** 발표한다. (개발 경험 개선, E2E 도입 등 숫자 기반 성과 포함)

## Action Items

### Phase 1 — 커뮤니케이션 & 회의 (회의실 들어가기 전 체크)

- [ ] **결론부터 (BLUF)**: "결론 → 이유 → 요청사항" 순서로 말했는가?
- [ ] **사전 조율 (Pre-alignment)**: 착수 전 "A방식으로 하려는데 동의하시나요?"를 먼저 물었는가?
- [ ] **비개발 용어 번역**: 기술 난이도를 '비용'과 '시간'으로 환산해 기획 단계에서 미리 전달했는가?
  - 예: "구현은 가능하지만 3일이 더 걸립니다. 일정 내에 하려면 제외하는 게 좋습니다."
- [ ] **회의 다이어트**: 회의 종료 시 '핵심 요약'과 'Action Item'을 직접 정리해 공유했는가?

### Phase 2 — 업무 수행 & 개발 (코딩 시작 전/후 체크)

- [ ] **단순화 질문 (KISS)**: 확장성 있는 구조가 꼭 필요한가? 하드코딩/단순 로직으로 빠르게 끝낼 수 있는가?
- [ ] **범위 방어 (Scope Control)**: 막판 수정 요청 시 "이번 차수 필수인가요? 다음 V2로 넘길까요?"라고 제안했는가?
- [ ] **마감 루틴 (Last Mile Check)**: 배포 직전, 내 수정이 다른 기능에 미칠 사이드 이펙트를 리스트업하고 체크했는가?

### Phase 3 — 태도 & 마인드셋 (매주 금요일 회고)

- [ ] **결정 내리기**: 이번 주에 결정을 미룬 일은 없었는가? (과도한 합의 지양)
- [ ] **감정 온도 조절**: 의견을 낼 때 공격적으로 보이거나, 반대로 너무 방어적이지 않았는가?
- [ ] **거시적 관점**: 내 코드 퀄리티보다 '프로젝트 전체 일정과 목표'를 우선순위에 두었는가?
